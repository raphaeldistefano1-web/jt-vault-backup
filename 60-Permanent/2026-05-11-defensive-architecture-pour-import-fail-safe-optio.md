Architecture pour `class-jt-migrate-import`: séparer les étapes en **critical** (interrompent l'import) vs **optional** (log warning, continue).

**Critical steps** (MUST NOT fail):
- Migration données core (posts, pages, users, bookings)
- Validation SQL + intégrité constraints
- Création tables/indices

**Optional steps** (CAN fail silently):
- Vérifications cross-table
- Cleanup données obsolètes
- Warm-up cache
- Hooks post-import

**Implémentation**: Try-catch granulaire par étape, log detailed errors, continue si fail.

**Rationale**: En migration multi-step, avoir 90% des données importées avec 2 warnings est mieux que 0% parce qu'une vérification optionnelle a échoué. Utilisateur peut investiguer warnings post-import, mais les données core sont sûres.

**Critère**: Pas de regression d'import behavior.
