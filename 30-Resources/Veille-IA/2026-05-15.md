# Synthèse du jour

La semaine qui s'ouvre n'est pas seulement technique — elle est structurelle. Deux signaux antagonistes traversent la sélection du jour : d'un côté, Anthropic acte une rupture tarifaire majeure en séparant l'accès programmatique du plan Max [[01]], forçant les builders et opérateurs solo à choisir entre un abonnement orienté IDE ou des crédits API distincts pour leurs automations ; de l'autre, Figure AI démontre des robots opérant en production warehouse sur des shifts complets de huit heures sans intervention humaine [[06]]. Ces deux trajectoires racontent la même histoire en sens inverse : les modèles texte stagnent en qualité perçue — plusieurs communautés techniques rapportent une dégradation silencieuse sur la génération frontend et le raisonnement — et l'accès se restreint, pendant que l'embodied AI franchit les portes d'usine. Pour les opérateurs avec un stack IA autonome, la conséquence pratique est immédiate : l'architecture doit absorber cette friction tarifaire. Ce n'est plus une considération périphérique à traiter lors d'un audit trimestriel — c'est un risque de pricing qui peut multiplier par dix le coût d'un workflow agent-heavy en quelques semaines, comme le rapportent les retours communautaires autour de la décision Anthropic [[01]]. La portabilité entre providers, longtemps bonne pratique, devient une assurance structurelle.

Sur l'architecture même des agents, la sélection expose un glissement de paradigme. Hermes Agent [[08]] a dépassé OpenClaw en tête des agents sur OpenRouter en moins de 90 jours — non pas en ajoutant des intégrations, mais en implémentant un cycle do-learn-improve : chaque tâche terminée génère automatiquement une skill markdown réutilisable, stockée en mémoire procédurale et retrouvable via SQLite FTS sans vector database externe. La mémoire est stratifiée en trois couches — session (travail courant), épisodique (sessions passées recherchables) et procédurale (skills réutilisables) — produisant un agent qui se renforce à chaque usage, précisément parce qu'il a mémorisé comment son opérateur travaille. En parallèle, Composio [[05]] apporte une réponse orthogonale au problème du context bloat : plutôt que d'exposer des centaines d'outils au modèle, trois outils de découverte sémantique permettent à l'agent de trouver le bon outil à l'exécution, avec une authentification centralisée permettant de basculer entre providers sans reconfiguration. Kimi K2.6 [[09]] pousse la logique jusqu'à l'orchestration massive — 300 agents simultanés sur un seul prompt — mais le principe clé reste identique : l'utilisateur décrit le résultat final, pas le workflow, et le modèle décompose et parallélise automatiquement. Les trois approches convergent vers le même diagnostic : la performance d'un système agentic dépend moins de la puissance du modèle que de la qualité de son architecture mémoire et de son accès aux outils.

Les défaillances prévisibles des systèmes agentics méritent une taxonomie rigoureuse avant d'aller plus loin dans l'automatisation. IBM Technology [[02]] documente les modes d'échec les plus fréquents : boucles infinies par absence de conditions de terminaison, hallucinations de planification où l'agent invente un plan cohérent mais irréalisable, et outils mal-confinés permettant des actions destructrices sans approbation explicite. La règle posée est opérationnelle — les agents échouent quand ils ont trop d'autonomie sans assez de contraintes, pas à cause du modèle lui-même. Les correctifs sont d'ordre ingénierie : max retries, max steps, max runtime imposés ; agent vérificateur indépendant entre planification et exécution ; principe du moindre privilège sur l'accès aux outils par tiers (read/write/delete). Dans la même veine, Santos et Faulkner [[04]] signent le diagnostic le plus radical de la sélection : le CI/CD traditionnel est mort à l'échelle agent. Quand des dizaines de PR s'ouvrent en parallèle, le goulot n'est plus la review humaine mais la latence de build — qui doit descendre sous une minute pour ne pas bloquer la boucle. L'architecture de remplacement est un harness à état persistant avec validation agent-à-agent, le humain n'intervenant que sur l'intent final, jamais sur le diff.

Deux vidéos plus pragmatiques ancrent ces architectures dans le quotidien. Rick Mulready [[03]] rappelle que les skills Claude sont fondamentalement des dossiers avec instructions — un pattern stable que les mises à jour ne cassent pas. Les sept skills qui font tourner son activité couvrent planification quotidienne, analyse vidéo et frameworks de décision ; tous construits en langage naturel via le Skill Creator, sans expertise technique. La méthode est reproductible : identifier les tâches récurrentes par cluster, construire une skill par cluster, combiner les skills en pipelines bout-à-bout. Deya [[07]] applique la même logique au niveau business : brain dump exhaustif des tâches récurrentes, scoring sur quatre axes (fréquence, ennui, impact, simplicité), puis délégation des processus stables à Claude pour générer des SOPs contextualisées avec étapes numérotées et règles d'escalation. L'ordre est invariable dans les deux cas : documenter d'abord, automatiser ensuite. Une automation sans documentation sous-jacente produit des nœuds noirs impossibles à auditer ou corriger quand un agent sort du périmètre — observation qui résonne directement avec le diagnostic IBM sur les boucles non-contraintes.

Pour le stack de Raphaël, ces signaux convergent en trois points de traction immédiats. Premier : le basculement Anthropic [[01]] touche directement OpenClaw, qui porte le gateway OAuth Codex, et les agents internes (code-explorer, pms-planner, pms-explorer) tournant via crédits Max — chiffrer le burn réel en tokens cette semaine est non-négociable avant que le pricing change ne crée une surprise de facturation, d'autant que les retours communautaires évoquent une multiplication par dix à quarante sur les workflows agent-heavy. Second : l'architecture do-learn-improve de Hermes [[08]] est applicable directement aux agents PMS qui traitent des workflows répétitifs (confirmations de réservation, requêtes guests) — une mémoire procédurale sur ces patterns réduirait précisément les hallucinations de planification que subit tout agent opérant sur une base de données de production. Le Vault Obsidian, avec ses 92 notes atomiques déjà indexées via MCP, constitue la base naturelle pour cette couche mémoire épisodique en SQLite léger, sans vector database externe. Troisième : le pipeline n8n qui porte les événements de réservations est l'endroit précis où implanter les conditions de terminaison strictes [[02]] : max steps, max retries, approbation explicite avant toute mutation en base production, séparation des agents par tiers de permission selon qu'ils lisent ou écrivent sur les tables métier.
