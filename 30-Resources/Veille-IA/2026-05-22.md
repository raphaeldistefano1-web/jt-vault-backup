# Synthèse du jour

La journée du 22 mai 2026 met en lumière une tension structurelle qui traverse désormais tout l'écosystème agentique : la pression commerciale des fondeurs de modèles converge avec les limites techniques réelles du raisonnement long-contexte, et c'est précisément dans cet étau que se jouent les décisions architecturales critiques des prochaines semaines. Mike Codeur [[01]] documente avec précision la bascule d'Anthropic : à partir du 15 juin 2026, l'usage programmatique des harnais tiers — Claude Code CLI, OpenClaw, Conductor, Cursor Z, OpenCode — sera plafonné à 200 dollars de crédit extra mensuel. Au-delà, tarif API pur, soit entre 2 000 et 3 000 dollars par mois pour un usage intensif quotidien. Ce n'est pas une hausse de prix déguisée ; c'est un mécanisme de recapture d'écosystème. Anthropic invoque la mauvaise gestion du contexte et du cache par les outils tiers pour justifier ce verrouillage progressif, mais la conclusion opérationnelle est limpide : tout stack qui dépend d'un harnais custom tiers entre dans une zone de risque économique qu'aucun praticien n'avait anticipée en début d'année. La deadline est dans vingt-quatre jours exactement.

Simultanément, deux analyses indépendantes d'Opus 4.7 arrivent à des conclusions qui se renforcent mutuellement. Craig Hewitt [[03]] identifie cinq changements comportementaux critiques : Opus 4.7 a abandonné la capacité d'inférence caractéristique de 4.6 — il ne remplit plus les blancs, n'internalise plus les instructions implicites, et n'auto-parallélise plus les sub-agents. La contrepartie est une rigueur d'exécution supérieure sur les tâches déterministes, à condition que le prompt soit exhaustif, exemplifié et explicitement borné en tokens. Platzi [[08]] pousse l'analyse plus loin avec des chiffres bruts : sur le benchmark MRCR (Multi-Round Context Recall), Opus 4.7 chute à 32,2% contre 78,3% pour 4.6 — une régression catastrophique sur la mémoire long-contexte. Combiné à une consommation de tokens en hausse de 35%, le calcul devient défavorable dès que la tâche mobilise un contexte riche ou une session longue. La leçon synthétique de ces deux sources : Opus 4.7 est un moteur de précision chirurgicale dans une fenêtre courte et explicite, pas un modèle de raisonnement étendu — une distinction qui change fondamentalement la façon dont on doit briefer chaque agent du pipeline.

Sur l'orchestration multi-agents, la sélection du jour aligne trois perspectives convergentes. Zen van Riel [[04]] pose la règle empirique la plus utile : quatre flux parallèles maximum — au-delà, la capacité de supervision mentale humaine se sature et la qualité de la revue s'effondre. Le message est contre-intuitif dans un contexte où les frameworks promettent des flottes illimitées, mais il correspond à ce que tout praticien honnête observe en production. Christoph Magnussen [[07]] introduit le concept d'Agent Drift — le glissement progressif et non-détecté d'un agent hors de son périmètre initial — comme le risque systémique le plus sous-estimé de tout setup multi-agents, avec comme conséquence directe une explosion des coûts. Sa recommandation : profiler d'abord chaque outil dans des terminaux parallèles indépendants avant de déléguer à un dashboard d'orchestration centralisé. Google, avec Antigravity 2.0 [[13]], tire la même conclusion architecturale depuis l'intérieur d'un hyperscaler : la décentralisation en sub-agents spécialisés avec project-level constraints surpasse l'agent monolithique surchargé — et cette philosophie est désormais encodée dans leur SDK. Le consensus du jour est sans ambiguïté : paralléliser, mais contrôler chaque flux avec des contraintes explicites, pas de la confiance implicite accordée au modèle.

La gestion de la context window s'impose comme le fil transversal invisible qui relie presque tous les contenus du jour. Matt Pocock [[02]] formalise le handoff pattern — distinct de la compaction classique — comme un mécanisme de branching de sessions : l'agent courant compresse ses apprentissages dans un fichier markdown structuré qu'un agent frais charge en début de nouvelle session, préservant ainsi la zone d'attention optimale sans diluer le contexte principal. Ce pattern est du sous-agent artisanal au sens littéral : il émule la délégation à coût quasi-nul, sans infrastructure MCP supplémentaire. La régression MRCR d'Opus 4.7 [[08]] rend ce pattern encore plus urgent : si le modèle ne retient que 32% des informations après plusieurs rounds, la stratégie défensive consiste à limiter la longueur des sessions et à ré-injecter le contexte critique en début de chaque fork. Roboverse [[14]] valide indirectement cette approche en conditions réelles : dans sa comparaison de cinq agents, la persistance de mémoire inter-sessions est le différentiateur décisif entre un outil utile une fois et un système qui s'améliore sur la durée.

Un axe plus opérationnel complète le tableau : l'automatisation navigateur et les workflows déterministes. Nate Herk [[05]] documente une combinaison Claude Code et Playwright CLI qui contourne les MCPs lourds en tokens pour des tâches d'automatisation web répétitives — la boucle test-erreur-apprentissage permet de cristalliser chaque flux réussi en skill réutilisable. L'argument économique est direct : Playwright CLI consomme significativement moins de contexte que le Chrome DevTools MCP, ce qui compte dans un environnement où chaque token est désormais capé ou facturé. Roboverse [[14]] confirme que n8n reste le meilleur choix pour les workflows événementiels déterministes — fiabilité supérieure aux agents chat libres, prévisibilité des coûts, et auditabilité complète des exécutions. Ce double verdict donne une carte d'allocation claire : ne pas demander à un LLM ce qu'un workflow visuel ferait mieux, et ne pas demander à un workflow visuel ce qui nécessite du jugement contextuel.

Pour le stack du PMS Le Jardin Tropical, ces signaux convergent en trois implications concrètes et indépendantes. La menace Plan Max [[01]] impose une mesure de consommation OpenClaw cette semaine — pas après le 15 juin, trop tard pour décider sereinement. Le handoff pattern [[02]], combiné à la règle des quatre flux parallèles, offre une discipline d'orchestration directement applicable au pipeline code-explorer vers code-architect vers implémentation vers code-reviewer : chaque agent produit son fichier de handoff, le suivant l'ingère proprement, et la context window du thread principal reste courte et efficace. Et la révision des prompts agents — pms-planner, ralph-loop, code-architect — pour ajouter des bornes de thinking explicites et des exemples concrets n'attend aucune décision stratégique : c'est un diff de configuration, pas une refonte. Ces trois ajustements sont indépendants et peuvent être menés en parallèle dans la journée. Le 22 mai n'est pas une journée d'annonces fracassantes, mais d'ajustements de cap sur un stack qui tourne déjà bien et qui mérite d'être blindé avant la vague.

## 02 — /handoff is my new favourite skill

- channel: Matt Pocock
- duration: 12 min
- published_relative: aujourd'hui
- url: https://www.youtube.com/watch?v=dtAJ2dOd3ko
- tools: Claude Code, GitHub, Codex, Copilot CLI, TLdraw SDK

**TL;DR** : Handoff skill compresses session context into markdown files for fresh agents, letting you split work across sessions while staying in the smart zone instead of hitting the dumb zone.

**Thèse** : Context windows have performance zones—early tokens get better attention, later tokens hit a "dumb zone." Handoff differs from compaction: it lets you branch off work to independent sessions and pass learnings back, creating a DIY sub-agent pattern that keeps each session focused instead of diluting it.

**Bonnes pratiques** :
1. Identify out-of-scope work mid-session and hand it off to a fresh agent session rather than extending the current session—keeps focus pure and prevents context dilution.
2. Always describe the purpose and focus of the handoff explicitly when creating the document, so the next agent knows what to prioritize.
3. Include a suggested skills section in handoff documents so the receiving session knows which workflows or tools to invoke without you having to repeat setup.
4. Save handoff files to your OS temporary directory, not your workspace—they're disposable, not documentation artifacts meant to persist.
5. Use pointers and references to existing artifacts (GitHub issues, design docs, code files) instead of duplicating content already captured elsewhere.

**Citation** : « It's almost like you've done a kind of DIY sub agent where you're able to use a context window for one specific task, compress your learnings from that task, and pass it back to the parent. »

**Applicable à ton stack** : Your multi-agent workflow (code-explorer → code-architect → implementation → code-reviewer) could adopt handoff patterns: each agent creates a markdown summary for the next, keeping the Vault clean and preventing context bloat across teams. Particularly useful when forking parallel work (prototype vs. spec, audit vs. implementation) without diluting the main session focus.
