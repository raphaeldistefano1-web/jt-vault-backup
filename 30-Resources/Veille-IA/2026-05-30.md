# Synthèse du jour

La leçon du jour n'est pas un modèle, c'est une thèse unifiée : ce qui détermine la performance d'un système IA n'est pas la version du LLM en son centre, mais la qualité du harness qui l'entoure. Trois vidéos distinctes, venues d'horizons différents, convergent sur ce même constat avec une précision remarquable. Fazt [[09]] le formule le plus clairement : le harness est le système d'exploitation de l'IA — context window, actions outillées, persistance structurée, boucles d'observation — et Opus 4.6 dans Claude Code surpasse Opus 4.8 dans VSCode+Copilot précisément parce que le premier dispose d'un meilleur harness. La chaîne AI随风 [[04]] reprend la même idée depuis l'angle de la documentation : ce qui amplifie l'agent, c'est l'ensemble de contraintes autour de lui — CLAUDE.md hiérarchisé par module, hooks de batching pour l'apprentissage continu, skills matérialisés en workflows déterministes — bien plus que le saut vers le modèle suivant. Erwan sur La MasterClass IA [[03]] referme la boucle : la bonne question n'est pas quel modèle est le plus puissant, mais quel modèle sert le mieux l'usage réel. La sélection du jour trace ainsi une ligne de crête entre deux erreurs symétriques : sous-investir dans le harness (vibe coding) ou sur-investir dans le modèle (benchmark chasing).

Sur le plan de l'architecture, la convergence la plus nette porte sur la modularisation de l'expertise IA. The Coding Gopher [[02]] expose le pattern Agent Skills avec une précision pédagogique rare : là où le god prompt antipattern noie les LLMs dans des dizaines de milliers de tokens de règles contradictoires, les skills versionnés en git opèrent par progressive disclosure — métadonnées légères d'abord (trente à cinquante tokens), instructions procédurales au besoin, références détaillées en deep retrieval uniquement. Chaque skill devient un répertoire autonome avec son dossier scripts/ pour la logique déterministe que le LLM n'a plus à halluciner, et son fichier skill.md pour l'orchestration. AI随风 [[04]] propose un complément immédiat : une Architecture Map dans CLAUDE.md qui liste les modules clés et leurs fichiers d'entrée, permettant aux agents de localiser le contexte pertinent cinq fois plus vite sans lire le projet complet à chaque invocation. Fazt [[09]] referme ce cercle avec le pattern d'orchestration hiérarchique : plusieurs agents en parallèle, chacun rapportant à un superviseur, chacun expert d'un périmètre délimité, reproduisant la logique d'une équipe engineering sans les latences humaines. Ces trois approches se superposent sans se contredire : elles décrivent le même invariant sous des angles différents.

Le troisième grand courant de la journée est une montée en puissance du strict engineering comme antidote systémique au vibe coding. Nuno Maduro [[10]] pose la thèse la plus directe : TypeScript strict ou PHP Stan activés sur l'ensemble du codebase, patterns documentés et appliqués systématiquement dans chaque fichier, couverture de tests à cent pour cent — non pas comme aspirations, mais comme contraintes CI bloquantes. L'argument pratique est convaincant : une IA entraînée sur un codebase cohérent apprend les conventions et les réplique dès la première itération, évitant les cycles de révision répétitifs. Tech With Tim [[01]] complète ce tableau par la couche sécurité : VibeSecure intégré en début de workflow force la détection de vulnérabilités avant production, là où la pratique habituelle les découvre après. IBM Technology [[05]] élève le raisonnement à l'échelle des architectures critiques : pour les chemins RGPD, financiers ou d'authentification, un seul agent qui hallucine avec confiance produit des erreurs indistinguables de la vérité — la solution n'est pas l'entraînement du modèle, mais l'architecture même du système, avec un agent générateur, un vérificateur, et un adversaire red-teaming avant chaque décision critique.

En marge de ces questions d'architecture logicielle, deux signaux infrastructure méritent attention. Zen van Riel [[08]] inverse une intuition répandue : dans les workflows agentiques modernes qui chainent recherche web, exécution Python, appels API et automation browser, le CPU est devenu le vrai bottleneck, non le GPU. L'inférence reste GPU-intensive, mais la tokenisation, l'orchestration des tool calls, la préparation des données entre chaque appel et les requêtes database consomment du CPU en quantités comparables, avec un ratio CPU:GPU qui s'approche de un pour un à un pour quatre sur certains pipelines de deep research. Profiler les deux simultanément avant de conclure à un besoin de GPU devient une pratique d'ingénierie, pas une option. Naval [[06]] pose un second signal : les modèles sont passés du niveau ingénieur junior au niveau principal, redistribuant les rôles — les agents héritent des tâches répétitives (boilerplate, documentation de conformité, tests), les humains se repositionnent sur l'architecture systémique et la vérification des décisions critiques. Documenter et mesurer, c'est désormais autoriser l'automatisation.

Rapporté au stack de Raphaël, l'ensemble de ces courants dessine une feuille de route cohérente. Le harness existant — Claude Code avec CLAUDE.md, hooks, agents internes, vault Obsidian comme base de contexte persistant — est déjà plus avancé que la majorité des configurations décrites dans ces vidéos. Mais plusieurs points aveugles ressortent clairement. Les agents internes du PMS (code-explorer, code-architect, code-reviewer) existent en tant que fichiers YAML, mais ne sont pas encore formalisés comme Agent Skills versionnés [[02]] avec progressive disclosure — chaque invocation charge probablement plus de contexte que nécessaire, et la logique déterministe reste dispersée dans CLAUDE.md plutôt qu'encapsulée dans des scripts autonomes. CLAUDE.md du PMS Le Jardin Tropical ne dispose pas encore d'une Architecture Map [[04]] listant les modules Frontdesk, Calendar, Booking, Payments avec leurs fichiers d'entrée — les agents recompilent cette carte mentalement à chaque session. Et les chemins critiques du PMS (paiement, NextAuth, données RGPD) n'ont pas encore l'architecture multi-agents vérificateur [[05]] justifiée par les enjeux réels : une violation RGPD ou une régression de paiement coûte bien plus que le token overhead d'un second agent de validation.

Le paradigme change donc : là où ChatGPT-2022 était GPU-dominant, les agents 2024+ doivent compter sur un ratio CPU:GPU plus proche de 1:1 à 1:4, et ignorer ce shift coûte en latence et scalabilité.

**Bonnes pratiques** :

4. Alloue CPU multi-core pour paralléliser les tâches agentic : un seul core saturé cache la vraie bottleneck ; mesure le pic total avant de conclure.
5. Pour workflows deep research (parsing 100+ documents), dimensionne CPU ≥ GPU ; pour thinking mode intensif (minutes de calcul), GPU reste critique — adapter l'infra au cas d'usage.

**Citation** : « Le changement est définitivement réel. On voit que les workflows agentic peuvent consommer beaucoup plus de temps CPU, mais il faut vraiment considérer ton cas d'usage. »

**Applicable à ton stack** : Sur ton infra agents Claude Code + n8n, profile CPU vs GPU avant de blâmer la GPU : tes agents code-explorer, code-architect qui chainient grep/read/appels API sont probablement plus CPU-limités que GPU-limités.
