# Synthèse du jour

La sélection du 3 juin s'articule autour d'un renversement de contrôle qui restructure l'ensemble des couches de l'outillage agent. Dynamic Workflows [[01]] — présenté par Mike Codeur comme le coup de grâce porté à CrewAI — incarne ce glissement : là où le développeur codait manuellement le coordinateur (dispatch, convergence, validation), Opus 4.8 prend désormais en charge cette responsabilité en temps réel. L'architecte ne programme plus la coordination, il décrit le problème et la structure de validation. Ce n'est pas une amélioration incrémentale d'une librairie d'orchestration : c'est un déplacement de couche — du code impératif vers le raisonnement délibératif du modèle lui-même. Trois autres vidéos gravitent autour de cette thèse par des entrées complémentaires : quel modèle choisir pour quelle tâche [[07]], quel harness amplifie réellement la puissance brute du modèle [[04]], quelle architecture de mémoire évite que l'agent reparte de zéro à chaque exécution [[05]]. La journée ne disperse pas — elle converge. L'unité de lecture de cette sélection est la question : comment construire un système agentic qui s'améliore dans le temps plutôt que de réexécuter les mêmes cycles à coût croissant ?

Sur l'orchestration proprement dite, les paramètres opérationnels sont posés avec précision [[01]]. Dynamic Workflows tourne sur 16 agents concurrents réels (1000 planifiés), ce qui impose une architecture en deux phases : phase 1 mapping et assignment, un agent par fichier ou module, puis phase 2 fix loop convergente vérifiée par deux reviewers internes automatiques qui réfutent chaque output. L'overhead d'initialisation est réel — le fast mode divise par trois la tarification mais le coût d'entrée reste élevé — ce qui rend le pattern inadapté aux tâches isolées ou aux petits bugfixes. La recommandation est calibrée : démarrer à dix agents, mesurer tokens contre gain de productivité, monter graduellement vers les limites. Cette rigueur s'articule directement avec la thèse portée par Tech With Tim [[04]] : la qualité d'un système agentic dépend moins du modèle brut que du harness — context management, qualité des tools, recovery des boucles d'erreur. Composer 2.5 de Cursor en est la démonstration empirique : quatorze fois moins cher qu'Opus 4.8 pour du code, trois fois plus rapide, avec une qualité comparable — non parce que le modèle est meilleur, mais parce que le harness de Cursor est supérieur. Le moteur et la voiture ne sont pas interchangeables.

La question de la sélection de modèle par tâche structure deux vidéos complémentaires. John Savill [[06]] la formule dans le contexte enterprise Azure Foundry : l'ère du modèle frontière universel est terminée. GPT 5.5 instant pour les flux à latence courte, Opus 4.8 ou GPT 5.5 thinking pour le raisonnement long-horizon, Deepseek V4 pour les cas sensibles au coût — chaque choix se justifie sur des critères mesurables, pas sur la nouveauté du modèle. Le même verdict émerge chez WorldofAI [[07]] en comparaison directe sur tâches concrètes : GPT 5.5 high reasoning domine l'engineering et les workflows multi-étapes complexes, Opus 4.8 excelle sur le design front-end et les tâches visuelles, Gemini 3.5 Flash sur l'itération rapide à budget contraint. La conclusion opérationnelle est identique dans les deux cas : évalue sur tes propres prompts et domaines, pas sur les benchmarks génériques dont les scores publics ne reflètent pas tes vrais besoins. Savill ajoute un pattern rarement explicité : le filtrage des données avant envoi à l'agent. Injecter un snapshot entier de contexte quand seul 10% est pertinent à la tâche en cours, c'est bruler des tokens et dégrader la qualité du raisonnement — la discipline d'alimentation du contexte compte autant que le choix de modèle.

Le second axe de la journée est moins spectaculaire que l'orchestration à mille agents, mais probablement plus structurant sur le long terme : la mémoire organisationnelle comme infrastructure critique. ByteMonk [[05]] part d'un constat sans détour — la connaissance qui vit dans une seule personne est morte dès qu'elle disparaît, et le coût réel ne réside pas dans le départ de l'expert mais dans le temps quotidien que toute l'équipe consacre à reconstruire le contexte dispersé entre Slack, GitHub, Jira et logs d'incident. Les agents actuels amplifient ce problème structurellement : chaque run repart de zéro, reconstruit les conventions, re-lit les mêmes runbooks. La solution proposée est architecturale et en trois couches — scope de tâche (contexte immédiat), scope d'équipe (conventions et runbooks), scope organisationnel (décisions globales) — avec accès scopé par rôle, coûts attribuables par équipe et trace expliquée de chaque action. Parker Prompts [[02]] valide cette thèse par l'expérience : Hermes Agent, accumulant des skills auto-générés sur un serveur persistent, démontre qu'un agent ayant atteint vingt skills ou plus complète les tâches similaires 40% plus rapidement, parce qu'il arrête de redécouvrir le processus à chaque exécution. La mémoire n'est pas un nice-to-have — c'est le multiplicateur de productivité le plus sous-exploité dans les architectures agents actuelles.

Pour le stack de Raphaël, ces axes convergent sur des décisions concrètes à chiffrer. Dynamic Workflows [[01]] est directement applicable aux migrations et audits massifs du PMS Le Jardin Tropical : remplacer le coordinateur codé en dur dans les workflows n8n event-driven par un orchestrateur piloté par Opus 4.8 répond exactement au pattern de la phase 2 fix loop convergente — test recommandé avec dix agents sur un chantier réel (migration Prisma, audit chambres) avant tout scaling. Le Vault Obsidian est déjà l'infrastructure documentaire — mais les agents internes (pms-planner, pms-explorer) reconstruisent encore le contexte décisionnel à chaque exécution [[05]]. Un loader de mémoire initiale pré-injectant les décisions vault pertinentes selon le scope hot/warm/cold réduirait structurellement le coût de re-contextualisation sur chaque incident de réservation. Le harness [[04]] est le levier transversal : optimiser le filtrage des données qui alimentent les agents du PMS — ne passer que le contexte vault pertinent à la tâche plutôt qu'un snapshot complet — produit plus de gains immédiats que basculer de modèle.

# À regarder en priorité

- [[05]] ByteMonk livre le cadre le plus actionnable pour le Vault Obsidian : mémoire organisationnelle en trois couches comme infrastructure critique. L'argument est quantifiable — 30 min économisées par incident — et le chantier loader pour les agents PMS internes est clairement défini.

Élimine la re-contextualisation manuelle à chaque incident de réservation — premier test sur pms-planner en mode urgence réservation.

5. Valide le ROI : fast mode divise par 3 la tarification mais overhead initial élevé — mesure coût réel vs gain productivité avant d'étendre à l'ensemble du pipeline.
