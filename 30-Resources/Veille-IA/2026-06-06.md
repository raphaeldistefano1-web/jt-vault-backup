# Synthèse du jour

La question centrale qui traverse la sélection du jour n'est pas quelle capacité brute choisir mais quel harness construire autour du modèle pour qu'il ne s'auto-évalue pas trop généreusement. Trois signaux convergents pointent vers le même diagnostic : un seul modèle qui planifie, implémente et se juge lui-même va structurellement produire un biais flatteur que seul un regard indépendant peut corriger [[01]]. Claude Code s'impose comme concurrent crédible non pas parce qu'il génère du meilleur code que Codex, mais parce qu'il fonctionne localement, avec les credentials réels, dans un modèle de confiance où l'utilisateur valide avant toute exécution [[07]]. OpenAI, en intégrant un éditeur web dans Codex avec hébergement Cloudflare [[08]], confirme que la compétition ne porte plus sur le modèle mais sur le harness — l'ensemble de features, connecteurs et intégrations qui entoure le modèle et retient l'utilisateur. Les modèles se commodifient en quelques semaines ; l'architecture qui les pilote, elle, crée la durée. C'est une bascule stratégique que la sélection du jour documente sur plusieurs angles complémentaires avec une précision opérationnelle rare.

Sur le front de la planification, le pattern documenté par Chase AI [[01]] introduit deux artefacts qui transforment la spécification en protocole vérifiable : un plan.md comme source de vérité finale, et un review_log.md qui enregistre tous les cycles de revue avec les corrections appliquées à chaque passe. La règle est stricte — on n'arrête pas tant que le reviewer indépendant n'a pas approuvé explicitement, et ce reviewer ne peut pas être le même modèle qui a produit le plan ; donner la mémoire complète des cycles précédents au reviewer évite les faux fixes et les remarques identiques qui reviennent en boucle. Ce principe d'indépendance s'articule directement avec la doctrine de design de skills qu'Anthropic formalise dans ses propres outils [[02]] : les skills efficaces ne prescrivent pas chaque étape mais documentent ce qu'il ne faut pas faire, structurés en dossiers multi-fichiers avec un fichier principal d'instructions et des sous-fichiers pour les anti-patterns, les edge cases et les exemples validés. La description d'un skill est écrite pour que le modèle découvre quand l'activer, pas pour rappeler à l'auteur son contenu. Ce design converge avec la structure SMB Router + SMB Onboard [[03]] du plugin Small Business d'Anthropic, qui démontre en 31 skills comment deux méta-skills de routing et de mémorisation contextuelle transforment un ensemble de compétences éparpillées en un système cohérent et auto-orientable.

La métaphore de l'OS n'est plus un slogan marketing. AI LABS [[05]] documente en production comment Claude Code coordonne des sous-agents en parallèle avec le déterminisme d'un scheduler kernel : le claude.md joue le rôle de registre central, les MCP servers sont les drivers, et les dynamic workflows constituent le mécanisme d'ordonnancement pour les tâches larges et indépendantes. La règle d'usage est précise et non négociable : les workflows sont efficaces pour des tâches parallélisables et coûteuses à vérifier, pas pour des chaînes séquentielles profondes qui nécessitent une boucle goal avec contexte partagé entre étapes. Sur des migrations de données, l'écart mesuré atteint 21 minutes contre plus d'une heure en séquentiel, soit un gain de 65% sur la latence d'exécution ; la condition technique critique est d'imposer un schéma JSON strict sur chaque agent pour forcer la convergence du résultat et éliminer les surprises de parsing. Julia McCoy [[07]] complète ce tableau en articulant pourquoi le modèle de confiance incrémental — l'agent prépare, l'humain valide, puis exécute — est la vraie compétence différenciante par rapport aux agents cloud, au-delà de toute comparaison benchmark.

Deux leviers d'optimisation des coûts token dominent la sélection, et leur combinaison est structurellement complémentaire. Graphify [[04]] transforme le repo en graphe de connaissance via trois passes déterministes — parsing code avec Tree-sitter, transcription audio, analyse sémantique des docs — que Claude interroge ensuite via graphify query au lieu de relancer des explore agents dynamiques. La réduction annoncée de 40 à 70% des coûts tokens vient du remplacement de l'exploration par la navigation sur une carte persistante, auto-reconstruite après chaque commit via graphify hook install sans aucun appel API supplémentaire ; c'est une économie structurelle qui s'accumule sur toutes les sessions plutôt qu'une optimisation ponctuelle. En parallèle, Henri ExplorIA [[06]] documente les cinq niveaux de raisonnement d'Opus 4.8 comme second levier de calibration : low pour les tâches de volume (enrichissement, classement), medium pour le compromis coût-qualité standard, high comme valeur par défaut recommandée, extra-high pour les agents autonomes longs, max réservé aux problèmes extrêmement complexes où surréfléchir devient contre-productif. La leçon qualitative d'Opus 4.8 est aussi importante que la calibration : le modèle a corrigé les hallucinations du 4.7 en priorisant l'honnêteté sur les benchmarks, ce qui rend la reconnaissance des propres limites du modèle un critère de sélection plus fiable que tout score standardisé.

Le mouvement local-first atteint une maturité opérationnelle que la sélection illustre concrètement. Jack Roberts [[10]] démontre que la stack Hermes Agent + Ollama + Qwen 3 30B livre une infrastructure 100% privée, à zéro coût API, avec un audit trail Ollama exploitable directement comme argument de compliance GDPR. Le trade-off est explicite et assumé : un an de retard sur les frontier models en échange de la propriété complète des données et de l'absence de rate limits sur les boucles background itératives à fort volume. La stratégie de routage préconisée n'est pas un binaire cloud ou local mais une segmentation par niveau de sensibilité : données clients, paiements et IP métier vers le vault local, traitements non sensibles vers le cloud pour la qualité brute. Pour les opérateurs solo avec des données hôtelières réglementées, cette architecture n'est pas une option avancée mais une exigence de base que les outils de 2026 rendent enfin accessibles sans infrastructure cloud dédiée — et le pipeline de confiance en couches qu'elle produit est directement compatible avec les agents locaux que Claude Code fait tourner sur terminal.

Pour le stack de Raphaël, ces courants se condensent en trois axes d'action immédiatement concrets. Sur le repo PMS Le Jardin Tropical, Graphify [[04]] est le levier de réduction tokens le plus rapide à activer : graphify . crée le graphe initial, graphify hook install le maintient à jour après chaque push sans coût API, et graphify query remplace les passes d'exploration coûteuses que pms-explorer et code-architect relancent à chaque session sur le même codebase. Sur la planification, le pattern adversarial [[01]] ferme le gap structurel entre les plans générés par code-architect et leur faisabilité réelle dans le contexte n8n event-driven : introduire un reviewer Codex indépendant avant validation de tout plan d'agent, avec un review_log.md persistant, transforme ce qui est actuellement un jugement auto-référentiel en protocole auditable. Sur n8n, le pattern SMB Router [[03]] propose un modèle de routing que les workflows event-driven du PMS répliquent déjà partiellement — le formaliser comme skill explicite avec contexte hôtel mémorisé (14 chambres, tarifs saisonniers, règles de réservation) réduirait la redondance des briefings ad hoc et libérerait du quota pour les décisions d'architecture qui en ont vraiment besoin.
